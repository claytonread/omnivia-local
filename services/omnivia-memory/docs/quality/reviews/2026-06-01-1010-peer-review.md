## Summary

The amendment fixes source record persistence in `ingest_file()`. Sources are now persisted via `source_repository.create()` when configured, chunks are linked to the persisted source ID, and duplicate ingestion by path is detected before re-processing. Overall the fix is well-structured and the implementation is correct.

**Recommendation:** Accept.

## Findings

### Low: Attribute name inconsistency in `__init__`

**Location:** `pipeline.py`, line 39

```python
self.chunker_repository = chunk_repository
```

The attribute is named `chunker_repository` but the parameter is `chunk_repository`. The field is only used once (lines 94-96) where it controls whether chunks are persisted. Given it refers to `ChunkRepository`, the parameter name `chunk_repository` is the correct one. The attribute name should be `self.chunk_repository` for consistency.

**Suggested fix:**
```python
self.chunk_repository = chunk_repository
```

This is a Low severity issue because the mismatch does not cause functional failures - the attribute works correctly within the class. However it is misleading for future maintainers.

---

### Low: `IngestResult` has no `__slots__` or equivalent

`IngestResult` is a `@dataclass`. Each time `ingest_file()` returns a result, a new object is allocated. For bulk directory ingestion this could add up, but this is not a correctness issue and the dataclass pattern is standard throughout the codebase.

**Recommendation:** Accept as-is for now. Reassess if profiling shows allocation pressure in `ingest_directory()`.

## What Was Verified

### Source persistence when `source_repository` is configured

**Location:** `pipeline.py`, lines 81-89

When `source_repository` is provided, the new `Source` record is created in memory (lines 72-79) then persisted via `self.source_repository.create(source)` (line 84). This is the correct order - content hash is computed by the extractor before the source object is constructed, so the source is always created with a valid hash.

The `create()` call can raise `ValueError` if a source with the same UUID already exists (race condition between the lookup at line 66 and the create at line 84). The except block at lines 85-89 handles this by fetching the existing record and using it instead. This is correct.

### Chunks linked to persisted source ID

**Location:** `pipeline.py`, line 92

```python
chunks = self.chunker.chunk(
    "" if result.content is None else result.content,
    source.id
)
```

The `source.id` used here is the ID of the in-memory `Source` object. If the race condition path was taken (lines 85-89), `source` is reassigned to the existing record from the database, so chunks will reference the correct persisted ID. If the normal path was taken, `source` is still the in-memory object with the same ID that was just stored. Either way, `source.id` is the correct value.

### Duplicate ingestion detection

**Location:** `pipeline.py`, lines 63-70

```python
if self.source_repository:
    existing = self.source_repository.get_by_path(str(file_path))
    if existing is not None:
        return IngestResult(source=existing, chunks=[])
```

This is a clean guard. When a source already exists, ingestion is skipped entirely and the existing source is returned with empty chunks. The return type is consistent (`IngestResult`), and no state is mutated. Correct.

Note: The check only runs when `source_repository` is configured. If `source_repository` is `None`, no duplicate check occurs and re-processing proceeds. This is the intended design based on the docstring at lines 44-48 and the absence of alternative deduplication logic. The test at `tests/test_ingestion.py`, line 1006 (`test_duplicate_ingestion_handled_predictably`) confirms this assumption and passes.

### Error handling for race conditions

**Location:** `pipeline.py`, lines 82-89

```python
if self.source_repository:
    try:
        self.source_repository.create(source)
    except ValueError:
        existing = self.source_repository.get_by_id(source.id)
        if existing:
            source = existing
```

The pattern is sound: try to create, and if the ID already exists, fetch the existing record. The condition `if existing:` prevents a scenario where `source` is set to `None` and used in `source.id` at line 92. The outer try/except at line 100 catches any other unexpected exceptions and returns an `IngestResult` with the error string, which is a reasonable fallback for a pipeline-level handler.

### Comment quality and docstrings

The module docstring at line 1 is generic. The `IngestResult` docstring (lines 16-21) is clear. The `ingest_file()` docstring (lines 42-49) explains the three key behaviors: source persistence, chunk linking, and duplicate detection. The inline comment at line 63 is helpful. The comment at lines 67-69 is slightly verbose but accurate.

One improvement opportunity: line 67-69 could be consolidated to a single comment:

```python
# Source already ingested - return existing source without re-processing
```

Overall the comments are well above the minimum standard.

Verified: `pipeline.py` imports only from `omnivia_memory.ingestion.*` and stdlib modules. No references to `external/reference/` exist in the file or in any transitively imported module within the ingestion package.

## Residual Risks

1. **Attribute name mismatch** (Low): `self.chunker_repository` should be `self.chunk_repository`. This is a maintenance risk only, not a correctness failure.

2. **No content-hash deduplication**: Duplicate detection uses path equality only. If a file is moved, re-ingestion creates a second source record. No test covers this scenario, and it may be acceptable given Phase 2 scope, but it is worth noting.

3. **Chunk repository parameter name mismatch**: `chunk_repository` passed to `__init__` but stored as `chunker_repository`. This is the same issue as finding #1, just in parameter form.
